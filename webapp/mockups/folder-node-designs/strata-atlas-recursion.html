SOFT vs HARD

Two expand actions, not one. Soft expand is filetree-style: the drawer entry opens to show its children as more list rows, inline, still inside the same drawer. Hard expand promotes the folder to a full graph compound node — a new boxed region with its own spatial body and its own drawer. The recursion question is: what happens when you soft-expand to depth 2 then hard-expand the deep entry? Below: the state model, the recursive anatomy, the scenario step-by-step, and the laws that fall out.

▶ collapsed (drawer row) ▼ soft-expanded (drawer row, children listed below) ▣ hard-expanded (graph compound node) □ unloaded indicator (any non-hard state) ↵ hard-expand · → soft-expand · ← soft-collapse · [−] seal · ⌘W demote
M
state model — four states per folder, three live transitions

Every folder in the system is in exactly one of four states. The state lives at the folder, not at the user — two windows looking at the same folder see the same state. Soft and hard are not nested levels of the same action; they're different actions on the same row.

cascade · hard-expanding a depth-N entry implicitly hard-expands every ancestor on the path to the nearest already-hard ancestor. The chain promotes together.
A
three states · recursive anatomy — drawer is the registry; graph is the projection

Every child of an expanded folder is in one of three states. The drawer is the canonical registry — every child has a row there, whether loaded or not. The graph body is a spatial projection of the loaded subset. Collapsed compounds keep their drawer row AND appear as sealed boxes in the body — so their edges to other graph nodes stay visible. That's the load-bearing trick that lets you see cross-folder relationships without expanding everything. Below: all three states demonstrated at L1, L2, and L3.

A ▶□ unloaded

Drawer row only. No graph projection — no node, no edges, no presence in any body. Just a row in the parent's registry.

Cheapest state. The project can have thousands of folders in this state with zero render cost.

in canvas: tasks-archive/ · README.md · BF-116/117/118
B ▶▣ collapsed · sealed

Drawer row AND sealed compound in body. The compound is a single boxed node — contents hidden, but the box itself participates in the graph: edges to and from its members are aggregated onto its border.

This is the new option you proposed. It lets users see cross-folder structure without paying load cost for the contents.

Drawer row AND open compound in body. The compound is itself an expanded folder — control bar + spatial body + its own drawer. Same anatomy as the outer one. Recursion.

The drawer row shows ▼ (matching the compound's expanded state) and an "↑ in body" mirror tag so the user knows the two views are the same entity.

L
laws — invariants that fall out
  1. Same anatomy at every level. A hard-expanded folder = control bar · spatial body · drawer. No exceptions, no special leaf level, no "outermost" case. Recursion is the design.
  2. Body holds loaded; drawer holds unloaded. A folder's children are partitioned exactly: ▣ hard children live as nodes/compounds in the body; ▶/▼ children live as rows in the drawer. A child is never both.
  3. Soft is free; hard cascades up. Soft-expand is purely local — composes freely at any depth. Hard-expand on a row at depth-N implicitly hard-expands every ancestor up to the nearest already-hard ancestor. Down does not cascade — only the explicit target promotes.
  4. Soft state survives hard transitions. If you soft-expanded openspec/ then hard-expanded it, the new L2 compound's drawer remembers your soft-expansion of its tasks/ child (if any). Closing and reopening preserves the unfold tree.
  5. Seal preserves; demote forgets. [−] keeps every nested compound, drawer state, and leaf node alive but hidden. ⌘W (demote) returns a compound to its parent drawer as ▶ collapsed and discards its internal state.
  6. Drawer rows can be focused; bodies can be focused; never both. Arrow-key scope is tied to one focus context at a time. Esc walks focus outward: drawer → parent compound → grandparent → canvas.
  7. The "1280-files cap" remains a hard ceiling on loaded count. Soft-expanding 5,000 list rows is fine — they're text. Hard-expanding 5,000 compounds is not. The drawer's job is let users browse without paying load cost.
E
edge cases — what to do when the model strains
all children loaded

If a folder has 0 unloaded children, the drawer disappears entirely — no empty drawer chrome. A small ×0 unloaded chip stays in the control bar to confirm there is no hidden frontier.

drawer too long

At >30 entries the drawer gets a fixed max-height and scrolls; a ⌘F filter input appears above the rows. Soft-expansion still works inside the scrolling view.

cross-folder edge to sealed

If a graph node in folder-nodes' body has an edge to BF-118 (which is now inside sealed openspec), the edge terminates at openspec's edge fan on its sealed border. Sealing never breaks reachability.

cycle via filetree

The filetree mirrors disk hierarchy — no cycles possible by construction. Symlinks resolve at load time: a symlinked folder loads as a graph leaf node with a link-to badge, never as a recursive compound.

same file open twice

A markdown file hard-expanded as a leaf is a single graph node. If two parent folders both have it loaded (legal in linked-projects), the node is shared across both compounds — Cytoscape's normal multi-parent rules apply.

depth >5

STRATA's hatching repeats at +45° / -45° / 90° / shading-stripe / dotted. Beyond depth 5, the deepest compound auto-focuses (zooms to fill the canvas) and ancestors collapse to a breadcrumb strip at the top.

D1?
revisit — with the recursion clear, is "filetree outer" back on the table?

Your clarification — "the compound graph would live within the filetree nodes; when a folder is expanded, that part of the filetree expands [into a graph]" — means D1-A is not the squashed-accordion thing I rendered before. It's the symmetric inverse of D1-B: filetree at the root, graph compounds bloom as the visualization of any hard-expanded folder.

With the recursive anatomy above, the two designs converge: at the lowest non-trivial scale they're the same thing. A hard-expanded folder has a control bar, a body of graph children, and a drawer of unloaded children — whether the outermost container is a 2D canvas (B) or a vertical scroll (A) only shapes the first frame you see when you open the app.

The real choice is: does the user enter the system through the canvas or through the filesystem? If you're a code-editor user (project opens to a tree, expand to see), A wins. If you're a graph-app user (project opens to a workspace, drill into territory), B wins. Both deliver the same recursion. Pick by what the first ten seconds of using the app should feel like.
