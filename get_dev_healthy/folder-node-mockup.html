Folder-node interaction — bug repro + proposed mockups

Bug (current dev-manu): hovering a folder compound shows a grab cursor on the entire folder body. You can drag the subgraph — good — but you cannot pan or right-click inside the empty folder area.

Proposal: drop manual folder repositioning entirely. Drag anywhere = pan the canvas. Add a small chrome handle whose only job is the collapse/expand chevron. The folder body becomes a normal pan/right-click surface. When collapsed, the handle is the folder node. Folder layout is left to the auto-layout pass.

Chip = node, always: the chrome chip is the folder's identity-as-a-node in both collapsed and expanded state — hover or click opens the hover editor showing the folder's note, identical to leaf nodes. The dashed body just appears around it when expanded. Same hover-zone close rules (leaves both the chip and the editor → editor dismisses).

Try each stage: drag anywhere to pan (including the folder body and the chrome) · right-click empty space for the canvas menu · click chevron to collapse/expand · hover any leaf or collapsed chip for the hover editor.

Trade-offs at a glance

Pick one — they all fix the bug, they differ in feel and screen cost.

Recommendation: ② top-left corner. The label already lives top-left in today's style, so the chrome replaces it without consuming new pixels. ④ is best if discoverability is weighted heavily; the vertical cost adds up fast when 4+ folders nest.

Universal contract across ②③④: drag = pan, always. There is no mouse gesture that repositions a folder or its subgraph — that's auto-layout's job. The chevron is the single non-pan target on the entire stage.
